# 信頼は持ち運べる

*Zenn で公開: <https://zenn.dev/xkumakichi/articles/e93a438265a682>*

*※ 筆者は XAIP プロトコルのメンテナです。本稿は設計の背景にある構造的論点を整理するもので、プロモーションではありません。*

---

**TL;DR** AI エージェントがツールを選ぶ瞬間は、同時に信頼の判断をしている瞬間でもある。そしてその判断の質は、「信頼データがどこから来ているか」でほぼ決まってしまう。もしそのデータが単一のゲートキーパー（レジストリ、プラットフォームのキュレーション、コミュニティのモデレーション）を通っているなら、エージェントはそのゲートキーパーの脆弱性まで丸ごと引き継ぐ。本稿は、AI エージェントのトラストインフラを provider-neutral かつ behavior-derived（振る舞い由来）で組むべき理由と、それを実装した場合の稼働データを扱う。

---

## ツール選択の問題

AI エージェントがタスクを受け取る。「React の hooks ドキュメントを取ってきて」。

プランナーが候補を並べる。ドキュメント系ツールが3つ、検索系が2つ、フォールバック用の Web スクレイパーが1つ。どれを選ぶ？

正直に言うと、いまのエージェントは「モデルの学習データに名前があるかどうか」と「プラットフォームが見せると決めたもの」で選んでいる。ランタイムの信頼シグナルが存在しないからだ。昨日どのツールが成功したか、どのツールが静かに古いデータを返しているか、どのツールが知らぬ間に deprecate されたか、何も知らないまま選んでいる。

つまり多くのエージェントは、信頼に基づいてツールを選んでいるのではない。**偶然うまくいった選択を再現しているだけで、失敗も同じ確率で再現している。**

これがツール選択の問題の正体で、本質は**信頼データの問題**だ。

## 信頼データが置かれうる3つの場所

ツールの信頼データは、大きく3つの異なる場所から来る。

1. **自己申告**。ツールの README に「これは良いツールです」と書いてある。
2. **プラットフォーム・キュレーション**。公開先のプラットフォームが「おすすめ」リストを持っている。
3. **振る舞い由来（behavior-derived）**。過去の実行が記録・署名・集約され、信頼値は主張ではなく結果から計算される。

ゲーミング、ドリフト、上流のポリシー変更に対して頑健なのは (3) だけだ。そして (3) を用意するのが一番面倒でもある。署名付きレシート、正規化された集約モデル、特定プラットフォームに依存しない identity システムまで要るからだ。

## なぜ provider-neutrality が構造的に重要か

仮に、ある単一コミュニティのレジストリの上にトラストスコアを構築したとする。

そのレジストリ自体が一つのトラストレイヤーだ。何を表示し、何を推奨し、何を削除するかを決めている。可視性ルールが変われば（あるツールを推奨する、他を下げる、参加を制限する）、スコア対象空間も暗黙に変わる。以前インデックスされていたツールが候補から消えることもあるし、そもそも登録できない作者のツールはレシートを1枚も積めない。この変化はツールの振る舞いについて何も語らない。ある時点でのレジストリの状態を反映しているだけだ。

これは特定のコミュニティを指して言っているわけではない。**上流の可視性判断が下流のトラストシグナルへ流れ込むタイプの階層システム**であれば、どこでも同じことが起きる。上流の判断は、意図するしないに関わらず、トラストモデルの暗黙の入力になってしまう。

> ポータブルなトラストレイヤーがなければ、エージェントはツールを選んでいない。決定を継承しているだけだ。

結論として、トラストインフラは**レシート・ID・スコアリングのすべてが持ち運び可能（portable）でなければならない**。コミュニティが消えても、データは照会可能であり続ける必要がある。プラットフォームがポリシーを変えても、スコアリングは動き続ける必要がある。ID プロバイダが消滅しても、エージェントは検証可能であり続ける必要がある。単一の上流に依存するトラストインフラは、トラストインフラではなく、**その上流の選好に対する脆弱なプロキシ**に過ぎない。

## portable なトラストレイヤーの姿

[XAIP](https://github.com/xkumakichi/xaip-protocol) はこの原則を実装したものの一つで、設計は構造的要件から出発している。

- **署名付きレシート（自己申告ではない）**。ツール実行はすべて Ed25519 で署名されたレシートを生成する: `{ agentDid, callerDid, taskHash, resultHash, success, latencyMs, timestamp }`。呼び出し側も co-sign するので、ツールが単独で自分の評判を水増しすることはできない。
- **標準ベースの ID**。エージェントと呼び出し側は [W3C DID](https://www.w3.org/TR/did-core/)（`did:key` / `did:web` / `did:xrpl`）を使う。プラットフォームのアカウントは不要。あるコミュニティで可視性を失ったエージェントも、他のどこでも同じ identity を保てる。
- **閾値ではなく Bayesian**。スコアは `bayesianScore × callerDiversity × coSignFactor` として計算する。DID 方式ごとに異なる prior を持つので、安価な identity は信頼を無料で得られず、高コストな identity も十分な証拠があれば同じスコアへ収束する。
- **Provider-neutral なレシート発行源**。[MCP](https://github.com/xkumakichi/xaip-protocol/tree/main/clients/claude-code-hook)、[LangChain.js](https://www.npmjs.com/package/xaip-langchain)、[OpenAI tool calling](https://www.npmjs.com/package/xaip-openai) の統合が**同じレシート形式**を出す。LangChain エージェントが作ったレシートと、OpenAI チャット completion のレシートは、バイトレベルで互換だ。トラストグラフは、エージェントがどのフレームワークで作られていようと一つのグラフになる。
- **自分で動かせる集約器**。リファレンス実装の Aggregator は小さな Cloudflare Worker（OSS）。公開インスタンスを信じたくなければ自分で動かせる。マルチ Aggregator のクォーラムは仕様に組み込んである。

## 稼働中のデータ

リファレンス・デプロイは数週間動いている。執筆時点で:

- スコア対象のツールサーバー **10台**（ドキュメント取得、推論、メモリ、ファイルシステム、検索、DB、VCS 他）
- 署名付き実行レシート **2,100 枚以上**
- CI による **毎日の自動収集**（毎実行で新しい caller 鍵を生成。caller diversity が一級のシグナル）

ライブダッシュボード: [xkumakichi.github.io/xaip-protocol](https://xkumakichi.github.io/xaip-protocol/)
Trust API: `https://xaip-trust-api.kuma-github.workers.dev/v1/servers`

今どのツールを選ぶべきか、その場で尋ねることもできる。

```bash
curl -X POST https://xaip-trust-api.kuma-github.workers.dev/v1/select \
  -H "Content-Type: application/json" \
  -d '{"task":"Fetch React docs","candidates":["context7","sequential-thinking","unknown-server"]}'
```

レスポンスには選択結果に加えて、**反実仮想（counterfactual）** が入っている。信頼データなしにランダムに選んだ場合どうなるか、というシミュレーションだ。ここが価値の核心で、信頼データはエージェントを無駄な呼び出しから救うか救わないか、そのどちらかにしかならない。

## provider-neutral であることが具体的に買うもの

- LangChain 製のエージェントと OpenAI SDK 製のエージェントが、**同じツールに関する信頼データを共有できる**。今日はできない。各フレームワークが独自の観測サイロを持っているからだ。
- **あるコミュニティで可視性を失った作者のツール**も、他のコミュニティの呼び出し側から信頼を積み続けられる。
- エージェントインフラ案件を審査する **grant レビュワー**が、特定プラットフォームのダッシュボードに依存せずレシートを独立検証できる。
- 将来の規制体制が「このエージェントのツール選択の根拠は何か」と問うたとき、**持ち運び可能で監査可能な答え**が用意されている。

## 次のフロンティア

仕様はオープンで、Aggregator は稼働していて、3 フレームワーク統合は npm 上にある。次の論点は **クラス別リスク評価**（class-aware risk evaluation）だ。外部台帳にアンカーされた settlement ツールは、自由消費される advisory ツールと同じ信頼シグナルで評価するべきではない。この論点は [v0.5 ドラフト](https://github.com/xkumakichi/xaip-protocol/blob/main/XAIP-SPEC-v0.5-DRAFT.md) で扱っている。

**AI エージェントのトラストインフラを、どの単一プラットフォーム・コミュニティ・モデレーターにも依存させてはならない**。上流の可視性判断が下流のトラストシグナルへ暗黙の入力として流れ込むからだ。これを portable なレイヤーとして早く構築するほど、エコシステムは、TLS 証明書やパッケージ署名についてすでにそうしているのと同じように、雰囲気ではなく数式でツール選択について推論できるようになる。
